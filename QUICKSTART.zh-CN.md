# VCDDD 快速上手

> 不需要了解 DDD，不需要看白皮书。按这份文档走，五步完成一个有设计的产品。

## 第二步：加载 skill

Skill 安装后，在 AI 对话开始时告诉它加载这个 skill：

```

```

```

```

## 第三步：跟着五步走

VCDDD 有五步，每一步都有固定的工作内容。你不需要完全搞懂每一步背后的原理，只需要配合 AI 回答它的问题，在每步结束时确认输出。

---

### V — 说清楚你想做什么

**你要做的事**：用自己的话说你想做什么，不用组织语言，想到什么说什么。

**示例**：

```
我想做一个面向国内民宿房东的管理工具。
房东现在用微信群、Excel 管订单太乱了。
我希望能有一个地方统一看到所有房间的状态、订单、
客人信息，还有收款记录。
```

**AI 会做什么**：把你说的话整理成结构化的 `input.md`，你确认一遍就行。

**你的确认动作**：

```
这个描述是准确的，确认。
```

---

### C — 和 AI 一起把模糊说清楚

**你要做的事**：AI 会问你一系列问题，你一一回答。这些问题的目的是把你头脑中的想法变成「白纸黑字的事实」。

**AI 可能会问**：

- 一个房东通常管几个房间？
- 订单从哪里来？（直接微信，还是 OTA 平台？）
- 收款是线上还是线下？退款怎么算？
- 如果两个订单撞了同一个房间，谁来负责发现这件事？

**你只需要如实回答**，不知道的就说不知道，AI 会帮你标记为「待确认」。

**结束时**：AI 会生成一份 `facts.md`，列出所有确认过的业务事实。你逐条看一遍：

```
第3条有误，房东不是按月结算，是每笔订单完成后即时结算。
其余都准确，确认。
```

**这步完成的标志**：你对 facts.md 里的每条内容都说了「确认」。

---

### D¹ — AI 设计系统边界（你来审）

**你要做的事**：这步主要是 AI 在干活。它会根据你确认的事实，拆分出系统的各个「域」（功能区），并说明每个域负责什么、不负责什么。

你不需要懂 DDD，只需要用常识判断 AI 的设计合不合理：

**AI 会给你看类似这样的东西**：

```
我识别出以下域：
1. 房间域：负责房间信息和状态管理
2. 订单域：负责订单的创建、变更和取消
3. 收款域：负责收款记录和退款
4. 客人域：负责客人信息

我的问题是：「冲突检测」（判断一个房间是否已被预订）
应该由谁来负责？是订单域，还是房间域？
```

**你来回答**：

```
我觉得应该由订单域负责，因为冲突是因为订单引起的。
```

AI 会继续推导，直到每个域都有清晰的设计文档。

**结束时**：你看一眼每个域的文档，觉得没有明显遗漏或错误，就确认：

```
域设计看起来合理，确认进入下一步。
```

---

### D² — 确定技术方案

**你要做的事**：告诉 AI 你要用什么技术栈来实现。

**示例**：

```
我们用 Flutter 做 App，后端用 Go，数据库用 PostgreSQL，部署在阿里云。
```

如果你不确定，也可以说：

```
前端我想要 Web 端，后端你帮我选，要能快速上线的方案。
```

AI 会把这些写成一份 `tech-stack.md`，作为后续所有代码的约定。你确认一遍：

```
技术选型确认，可以开始写代码了。
```

---

### D³ — 开始写代码

**你要做的事**：现在可以开始叫 AI 写代码了。

但和普通「直接让 AI 写代码」不同的是，现在代码的每一个决定都有设计文档作为依据。你在叫 AI 实现某个功能时，AI 会自动对照设计文档，不会乱写。

**示例**：

```
先实现订单域的核心逻辑，包括创建订单和冲突检测。
```

如果 AI 在实现过程中发现设计文档里有缺漏，它会告诉你：

```
我发现 facts.md 里没有定义「订单超时未支付」的处理方式，
需要先澄清这个边界才能继续实现支付流程。
```

这时候你回头补充事实，AI 再继续。

## 常见问题

**Q：我不懂 DDD，可以用吗？**

可以。这个 skill 设计的初衷之一，就是让不懂 DDD 的人也能得到 DDD 的好处。你只需要把你的业务想清楚，技术设计的部分 AI 来做。

**Q：每一步都必须做吗？**

原则上是的，而且顺序不能跳。跳过「澄清事实」直接去「设计域」，会导致设计建立在假设上；跳过「域设计」直接写代码，会导致代码没有业务边界的约束，越写越乱。

**Q：已经有一个在跑的项目，可以用吗？**

可以，但第一次跑时要从 V 步开始「补课」——把现有系统的业务真相整理成 facts.md，再评审现有代码是否和这些事实一致。这比新项目多一个「摸清现状」的动作。

**Q：AI 设计的域我看不懂怎么办？**

直接说：

```
这个域设计我看不明白，能用更简单的话解释一下吗？
特别是「协作契约」那部分。
```

AI 会用非技术语言重新解释。你有权要求 AI 把每一个设计决策用业务语言说清楚，直到你能理解并做判断为止。

**Q：一个步骤能做多久？**

没有时间限制。V 步可能 10 分钟就完成，C 步对于复杂产品可能需要几轮对话。重要的是每一步结束时你真正确认了那份文档，而不是走过场。

## 最小完整示例

```
用户：加载 vcddd。我想做一个团队内部的假期审批工具。
      员工提交申请，直属上级审批，HR 看汇总报表。

AI：[进入 V 步] 我来整理一下你的意图...
    [生成 input.md]
    请确认这份意图描述是否准确？

用户：准确，确认。

AI：[进入 C 步] 我有几个问题要澄清...
    1. 一个员工可以同时提交多个假期申请吗？
    2. 审批拒绝后，员工可以修改再提交吗？
    3. 哪些假期类型需要区别对待（年假/病假/事假）？

用户：1. 可以，但同一时间段只能有一个。
      2. 可以修改，但只能修改一次。
      3. 都需要，年假有余额限制，其他两种没有。

AI：[整理成 facts.md，共 11 条事实]
    请逐条确认...

用户：第7条不对，病假不需要提前申请，可以事后补录。

AI：[更新 facts.md]
    更新完成，请再次确认...

用户：确认。

AI：[进入 D¹] 根据以上事实，我识别出3个域：
    申请域、审批域、额度域...
    [生成域设计文档]

用户：[审阅后] 确认，进入技术选型。

AI：[进入 D²] 你打算用什么技术栈？...

用户：React + Node.js + PostgreSQL。

AI：[生成 tech-stack.md，确认后进入 D³]

用户：先实现申请域，员工提交假期申请的完整流程。

AI：[开始写代码，所有命名和逻辑严格对应 facts.md 和域设计]
```

## 推荐初始 Prompt

把以下内容粘贴给 AI，作为一次 VCDDD 工作会话的开场：

```
请加载 vcddd skill。

我要用 VCDDD 方法设计一个新项目，请从 V 步开始引导我。
我对 DDD 不熟悉，请用业务语言而不是技术语言来解释每一步。

我的项目描述如下：
[在这里写你的想法，想到什么说什么，不用很正式]
```
