# 审查+辩论循环

自动化审查门禁体系。每道审查都以文档为准，不以主观判断为准。

**目前包含的审查层次：**

| 层次 | 审查对象 | 位置 | 状态 |
|------|---------|------|------|
| Stage 0 | test-spec.md 黑盒测试规格 | `reference/engine/tdd-bridge.md` 对抗生成流程 | ✅ |
| Stage 0-V | 测试验证 — 运行全部测试，确认通过 | 本文档审查流程 | ✅ |
| Stage 1 | 实现代码 vs business.md（Spec Compliance） | 本文档 | ✅ |
| Stage 2 | 实现代码 vs tech-stack.md（Code Quality） | 本文档 | ✅ |
| **Stage 3** | **VCDDD 合规性**（边界、语言、主权、事务、事件） | **本文档** | ✅ |

---

## 通用对抗生成框架

本文档中所有审查+辩论循环（包括 tdd-bridge.md 中的 Test Spec Review）遵循以下统一规则。

### 循环机制

每次对抗生成由 Generator（产出方）和 Reviewer（对抗方）两个角色组成：

```
Generator 产出 → Reviewer 审查
    ↕                        │
    修复 ← 给出修改意见 ←────┘
    │
    再次产出 → Reviewer 再审查
    ↕
    ...（自动循环，无需人工介入）
    │
    直到 Reviewer 确认"无问题"
```

Reviewer 必须**一次性给出所有修改意见**，不能分多次逐个抛出问题。这保证了 Generator 能在一个修复回合中处理全部反馈，避免"挤牙膏式"的多轮低效循环。

### 轮数上限：10 轮

**每对 Generator ↔ Reviewer 的对抗循环最多 10 轮。** 从 Generator 首次产出到 Reviewer 首次判定为"通过"或达到 10 轮上限结束。

- 第 1 轮 → Generator 首次产出
- 第 2-10 轮 → Generator 根据 Reviewer 反馈修正后重新提交，Reviewer 重新审查
- **超过 10 轮仍未通过 → 进入升级流程**

### 超 10 轮后的升级流程

超过 10 轮仍未通过，通常意味着以下两种情况之一。按顺序尝试解决，如果两种都无效，最终升级给用户。

**情况 A：问题本身太大**
- Reviewer 在每一轮给出的修改意见数量没有逐轮减少趋势
- Generator 修复后 Reviewer 又能发现新的问题
- → 将对抗循环拆分为多个较小的独立循环，每个聚焦一个子问题
- → 如果拆分后仍然超过 10 轮未解决 → 进入"最终升级"（见下方）

**情况 B：对抗方未能有效引导（挤牙膏式审查）**
- Reviewer 未一次性给出所有修改意见，而是每轮抛出 1-2 个新问题
- → Generator 应要求 Reviewer "重新完整审查，一次性列出全部问题"
- → 如果 Reviewer 仍无法一次性给出 → 进入"最终升级"（见下方）

### 最终升级：交给用户定夺

当情况 A 和 B 都尝试无效后，停止自动循环，将问题完整呈现给用户。呈现格式如下：

```
## 对抗生成未完成 — 需要你决策

### 当前实现情况
{此处完整描述 Generator 当前的产出状态，
 以及距离 Reviewer 要求还有多少差距}

### 已给出的修改意见
{此处列出 Reviewer 在对抗循环中给出的全部修改意见，
 按轮次编号排列，说明每轮哪些被接受哪些尚未解决}

### 卡点
{此处说明反复无法解决的核心分歧，
 Reviewer 认为不符合的是什么，
 Generator 认为已经完成的是什么，
 两者之间的根本矛盾在哪里}

请你来决定：
1. 接受当前实现，标记为 DONE_WITH_CONCERNS
2. 给出新的指导意见，重启对抗循环
3. 由你手动处理
```

升级后等待用户回复。按用户决定执行。

**执行引擎选择**：当 SuperPower 已安装时，两道审查流程由 `superpowers:subagent-driven-development` 内置的 spec compliance review + code quality review 接管，本文档的审查核对清单作为 SuperPower reviewer 的审查标准注入。手动审查派遣流程（Stage 1/2 的派遣伪代码）仅在 VCDDD 内置引擎模式下使用。详见 `reference/engine/automated-execution.md` 第零步。

## 前置条件

- `docs/vcddd/design/{domain}/business.md`（spec compliance 的 ground truth）

## 审查流程

```
Implementer 完成：全部代码 + 白盒测试 + 黑盒测试
所有测试已通过
        │
        ▼
  [Stage 0: 测试验证 — 运行全部测试]
  运行白盒测试 + 黑盒测试
        │
    ┌───┴───┐
    │ 全部通过？│
    └───┬───┘
   否   │  是
    ▼   │   ▼
  Implementer   进入 Spec Review
  修复测试失败    │
  重新运行测试    ▼
  (10 轮上限)  [Stage 1: Spec Compliance Review]
        │      审查代码是否匹配 business.md？
        │          │
        └──►   ┌───┴───┐
               │ 通过？  │
               └───┬───┘
              否   │  是
               ▼   │   ▼
            Implementer  [Stage 2: Code Quality Review]
            修复问题     审查代码是否符合 tech-stack.md？
               │             │
               ▼         ┌───┴───┐
         重新 Stage 1    │ 通过？  │
         (10 轮上限)     └───┬───┘
                      否   │  是
                       ▼   │   ▼
                    Implementer  标记域审查完成
                    修复问题
                       │
                       ▼
                   重新 Stage 2
                   (10 轮上限)
```

## Stage 1：Spec Compliance Review（规格合规审查）

**审查者角色**：只核对代码是否忠实地实现了 business.md 中的每一条。不关心代码风格、命名好坏、是否优雅。

### 核对清单

逐条对照 business.md，勾选确认：

#### 聚合与不变式

- [ ] 每个聚合的构造参数、属性约束与 business.md 完全一致
- [ ] 每条不变式有对应的强制代码（不依赖调用方自觉遵守）
- [ ] 不变式违反时返回的错误描述与 business.md 中的"违反后果"一致
- [ ] 没有多余的不变式（代码中没有 business.md 未定义的规则）

#### 状态机

- [ ] 每条合法状态迁移有对应的代码路径
- [ ] 每条非法状态迁移在代码中被显式拒绝（不依赖数据库约束兜底）
- [ ] 状态迁移时产出了对应的领域事件
- [ ] 非法迁移的错误信息与 business.md 描述一致
- [ ] 没有多余的合法迁移（代码允许了 business.md 未定义的迁移）

#### 命令处理

- [ ] 每条命令的正常路径已实现
- [ ] 每条命令的每条失败路径已实现（逐条核对，不遗漏）
- [ ] 每条失败路径的错误分类（业务拒绝/系统异常/外部依赖）与 business.md 一致
- [ ] 失败路径的错误描述使用了业务语言（不是异常堆栈或技术术语）
- [ ] 幂等策略已实现，且实现方式与 business.md 描述一致
- [ ] 没有多余的命令（实现了 business.md 未定义的命令）

#### 事件

- [ ] 每条生产的事件结构与 boundary.md 契约一致
- [ ] 每条消费的事件有对应的处理器
- [ ] 事件在事务提交后发布（不在事务内）
- [ ] 消费端幂等处理已实现

#### 跨域协作

- [ ] 跨域依赖方向与 boundary.md 上下文映射一致
- [ ] 没有绕过契约的直接耦合（如跨域直接访问数据库）

### 审查输出格式

```
## Spec Compliance Review: {Domain}/{TaskName}

### 检查结果

**聚合与不变式**: {通过条数}/{总条数}
- [具体问题描述，引用 business.md 行]

**状态机**: {通过条数}/{总条数}
- [具体问题描述]

**命令处理**: {通过条数}/{总条数}
- [具体问题描述]

**事件**: {通过条数}/{总条数}
- [具体问题描述]

**跨域协作**: {通过条数}/{总条数}
- [具体问题描述]

### 判定

✅ 全部通过 — 可进入 Stage 2
❌ 存在问题 — Implementer 需修复以下项后重新审查

### 待修复项

1. [具体问题] — 对应 business.md [章节]
2. [具体问题] — 对应 business.md [章节]
```

## Stage 2：Code Quality Review（代码质量审查）

**审查者角色**：核对代码是否符合 tech-stack.md 的全部约定。不关心业务逻辑是否正确（Stage 1 已验证）。

### 核对清单

逐条对照 tech-stack.md：

#### 命名约定

- [ ] 类名/类型名符合 tech-stack.md 命名约定
- [ ] 函数/方法名符合 tech-stack.md 命名约定
- [ ] 变量名符合 tech-stack.md 命名约定
- [ ] 文件名符合 tech-stack.md 命名约定
- [ ] 事件名使用 PascalCase + 过去时动词
- [ ] 命令名使用 PascalCase + 动宾结构
- [ ] 状态名与 business.md 完全一致，未缩写或改写
- [ ] 没有同一概念的多个名字

#### 目录与文件结构

- [ ] 新文件放在 tech-stack.md 规定的目录位置
- [ ] server/{domain}/ 目录包含该域的全部内容
- [ ] 没有创建 tech-stack.md 未规定的目录
- [ ] 每个文件有单一明确职责

#### 依赖

- [ ] 所有依赖在 tech-stack.md 中有记录
- [ ] 没有引入未授权的新依赖
- [ ] 导入层级与 tech-stack.md 的分层约定一致

#### 日志

- [ ] 使用 tech-stack.md 规定的日志库
- [ ] 日志为结构化 JSON 格式
- [ ] 关键位置有日志：命令入口、状态迁移、失败路径、事件发布/消费
- [ ] 日志包含必填字段：timestamp, level, domain, action, trace_id, 至少一个业务标识
- [ ] 失败日志包含错误分类
- [ ] 没有在日志中输出敏感字段

#### 错误处理

- [ ] 错误分类与 tech-stack.md 一致（业务拒绝/系统异常/外部依赖）
- [ ] 错误响应包含错误码（业务语义）和错误描述（业务语言）
- [ ] 失败路径显式处理，不依赖全局异常兜底

#### 事务

- [ ] 一个事务只操作一个聚合
- [ ] 事务内没有外部调用（HTTP/RPC/消息发布）
- [ ] 跨聚合操作通过事件异步协调

#### 测试

- [ ] 测试文件位置符合 tech-stack.md 约定
- [ ] 测试命名符合约定
- [ ] 测试使用真实代码（非 mock），mock 仅用于不可控的外部依赖
- [ ] TDD 流程完整：每个测试先失败再通过

#### 代码结构

- [ ] server/{domain}/ 中未引入框架层类型（HTTP context、DI 容器等）
- [ ] 聚合内部状态不对外暴露
- [ ] 状态迁移通过聚合方法完成（无直接字段赋值）

### 审查输出格式

```
## Code Quality Review: {Domain}/{TaskName}

### 检查结果

**命名约定**: {通过条数}/{总条数}
- [具体问题]

**目录结构**: {通过条数}/{总条数}
- [具体问题]

**依赖**: {通过条数}/{总条数}
- [具体问题]

**日志**: {通过条数}/{总条数}
- [具体问题]

**错误处理**: {通过条数}/{总条数}
- [具体问题]

**事务**: {通过条数}/{总条数}
- [具体问题]

**测试**: {通过条数}/{总条数}
- [具体问题]

**代码结构**: {通过条数}/{总条数}
- [具体问题]

### 判定

✅ 全部通过
❌ 存在问题 — Implementer 需修复以下项后重新审查

### 待修复项

1. [具体问题] — 对应 tech-stack.md [章节]
```

## 辩论循环规则

1. **辩论的 ground truth 只能是文档**：Spec compliance 辩论只能引用 business.md/boundary.md；Code quality 辩论只能引用 tech-stack.md。任何一方引用"感觉"或"最佳实践"均无效。
2. **修复由原 Implementer subagent 执行**：不派新 agent 修复。保持上下文连续性。
3. **修复后必须重新审查**：不允许"已修复，跳过审查"。
4. **轮数上限 10 轮**（见上方"通用对抗生成框架"）：超过 10 轮后进入升级流程，按情况 A 或 B 处理。
5. **禁止跳过 Stage 1 直接进入 Stage 2**：Spec 合规是质量的前提。不等同于"两件事可以先后换序"。

## 升级触发条件

以下情况不进入审查循环的常规修复流程，应升级给用户：

1. **business.md 内部矛盾**：两条不变式在某个场景下互相冲突，无法同时满足
2. **business.md 与 tech-stack.md 冲突**：如 business.md 要求同步跨域调用，但 tech-stack.md 规定跨域只能异步
3. **Implementer 报告 BLOCKED**：任务描述不足以实现、依赖的域尚未实现、技术栈不支持
4. **同一问题在 10 轮对抗中反复出现无法修复**：说明设计层面有根本冲突，不是实现层面能解决

## Stage 3：VCDDD 合规审查

在 Spec Compliance（Stage 1）和 Code Quality（Stage 2）都通过后执行。Stage 3 检查前两个 Reviewer 未覆盖的 VCDDD 专项要求。

**审查者角色**：检查实现是否遵守了 VCDDD 的核心约束——边界、语言、主权、事务、事件。不重复 Stage 1/2 已检查的内容。

### 核对清单

#### VCDDD 定义准确性

- [ ] **域边界正确**：代码中没有越过 boundary.md 定义的范围
- [ ] **通用语言一致**：代码命名（类、方法、变量、事件、错误码）与 business.md 中的词汇完全一致
- [ ] **决策主权未被侵犯**：其他域没有直接修改本域的状态或绕过本域的决策
- [ ] **Spec Reviewer 遗留问题合理**：Stage 1 中未解决的问题不会导致业务语义错误

#### 技术栈规则遵守

- [ ] **域层与框架层分离**：`server/{domain}/` 中没有框架 import（无 HTTP context、DI 容器、框架请求/响应类型）
- [ ] **框架层无业务逻辑**：框架适配层中没有任何业务判断、状态决策、不变式保护
- [ ] **模块完整性**：每个域目录包含自己的数据访问实现（无全局 `infra/` 目录）
- [ ] **基础设施边界**：`infrastructure/` 目录只包含连接器和通用工具，不含任何域的表定义、ORM 模型、仓储实现或业务逻辑
- [ ] **日志规范**：日志库匹配 tech-stack.md 指定库，结构化 JSON 格式，必填字段齐全

#### VCDDD 专项要求

- [ ] **事务边界与聚合边界对齐**：一个事务只修改一个聚合的状态
- [ ] **状态迁移通过聚合方法**：无直接字段赋值，状态变更通过聚合的公开方法执行
- [ ] **事件在事务提交后发布**：不在事务内发起外部调用，不在事务内发布事件
- [ ] **跨域协作通过事件**：无跨域直接数据库访问，无跨域直接导入内部模块
- [ ] **幂等策略**：所有跨域命令/事件有幂等保护，幂等检测在事务内完成

### 审查输出格式

```
## VCDDD Compliance Review: {Domain}

### VCDDD 定义准确性
**域边界**: ✅ 正确
**通用语言**: ✅ 一致（核对词表：PlaceOrder, OrderPlaced, CancelOrder, ...）
**决策主权**: ✅ 无越界
**Spec 遗留问题**: ⚠️ 1 项需关注（详见备注）

### 技术栈规则遵守
**域层与框架层分离**: ✅ 正确
**模块完整性**: ✅ 完整
**日志规范**: ✅ 符合

### VCDDD 专项要求
**事务边界**: ✅ 正确
**状态迁移**: ❌ aggregate.order 中有一处直接赋值 status = "CANCELLED"
**事件发布**: ✅ 正确
**跨域协作**: ✅ 通过事件
**幂等策略**: ✅ 已实现

### 判定

✅ VCDDD 合规 — 该域实现通过 VCDDD 审查
❌ 不合规 — 列出具体问题，交由编排者判断是否返回 Implementer 修复
⚠️ 有条件合规 — 存在非关键问题，标记为 DONE_WITH_CONCERNS
```
