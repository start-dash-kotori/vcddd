# 域设计线

域设计线的职责是基于已确认的业务事实，推导域结构：边界、不变式、状态机、协作契约。

## 前置条件

- SKILL.md 中的理论基座必须已内化
- `docs/vcddd/requirements/facts.md` 必须存在且经用户确认，否则不能进入域设计

## 入口判断：首次运行 vs 增量更新

进入域设计线的第一件事，判断当前模式：

```
docs/vcddd/design/ 目录下是否已有域文档？
    │
    ├── 没有（首次运行）
    │     → 从第一步开始，完整走完全部步骤
    │
    └── 已有（增量更新）
          → 读取已有的 boundary.md 建立当前域全景
          → 判断本次 facts.md 的变化影响哪些域
          → 只对受影响的域重新推导，不重跑整个设计
          → 写入时以"修正/扩展"方式更新，不替换无关域的文档
```

## 域设计线目标

1. 以 `docs/vcddd/requirements/facts.md` 为输入，不依赖对话历史
2. 推导域结构：哪些域需要新建、哪些需要修改、边界如何划分
3. 输出不变式和状态机，不是接口列表
4. 经用户确认后写入域文档

## 禁止性规则

- `facts.md` 未经用户确认前，不进入域设计
- 域设计未经用户确认前，不写入域文档
- 不主动进入实现，不提示用户写代码

---

**输入**：`docs/vcddd/requirements/facts.md`（经用户确认，不依赖对话历史）

**输出**：`docs/vcddd/design/{domain}/boundary.md` + `business.md`（经用户确认后写入）

---

## 第一步：建立全局视图

读 `facts.md`，在脑中建立：
- 有哪些业务事实，分别属于什么性质（概念/状态/约束/过程/协作）
- 现有域文档（如果已有）：`design/{domain}/boundary.md`
- 当前域全景：哪些域已存在，各自的决策主权是什么

---

## 第二步：识别域边界

### 追问链

对每一组业务事实，按顺序追问：

```
这组事实，是否有独立的业务语言？
    → 是：同一个词在这里有特定含义，在其他地方含义不同
    → 否：归入已有域

这组事实，是否有独立的不变式？
    → 是：约束只在这个范围内成立，与其他域的约束不互相依赖
    → 否：考虑合并

这组事实，是否需要独立演进？
    → 是：未来可能单独变化，变化不应影响其他域
    → 否：合并更合适
```

**三个问题都是"是"，才有独立成域的理由。** 只满足一两个的，优先归入已有域。

### 边界模糊时的消解顺序

1. 先问：现有域的 `boundary.md` 能不能承接？边界描述不够准确时，先扩展已有域，不急于新建
2. 仍然无法承接：追问"如果两个地方都在做这个判断，会发生什么"——如果答案是语义冲突，必须拆开
3. 主权归属有两种合理方案时，向用户提问，给出两个选项及各自含义

---

## 第三步：判断变化性质

对每个受影响的域，判断这次变化的性质：

| 变化性质 | 判断标准 | 处理方式 |
|----------|----------|----------|
| **能力扩展** | 原有边界内新增命令/事件/读模型，不变式和状态机核心路径不变 | 更新 boundary.md |
| **业务扩展** | 不变式或状态机有新增，但原有路径不破坏 | 更新 boundary.md + business.md |
| **边界调整** | 原有决策归属发生变化，或协作契约需要重新定义 | 全量审查受影响域 |
| **新建域** | 出现全新的、现有域无法承接的决策主权单元 | 新建域文档 |
| **合并/拆分** | 现有边界划分与业务事实不符 | 重组域结构，高风险，需用户确认 |

**边界调整和合并/拆分是高风险操作**，必须向用户明确说明影响范围，再确认。

---

## 第四步：设计不变式和状态机

域设计的核心输出是不变式和状态机，不是接口列表。

### 不变式设计

对每条候选不变式，追问：

```
这条约束必须立即成立，还是可以延迟收敛？
    → 必须立即：进入聚合，作为强制不变式
    → 可以延迟：通过事件异步修正，不进聚合

违反这条约束，后果是什么？谁来承担？
    → 没有明确后果：这条约束的边界没想清楚，追问用户
    → 有明确后果：记录违反后果，作为不变式描述的一部分
```

### 状态机设计

对每个需要状态化的过程：

```
状态集合是什么？（列举所有合法位置）
每个状态允许什么动作？（合法迁移）
明确禁止哪些迁移？（非法迁移，防止误操作）
失败停在哪里？能否恢复？（失败语义）
```

**必须同时设计合法迁移和非法迁移**——非法迁移是防止实现层误操作的关键约束。

---

## 第五步：设计协作契约

域间协作只有两种方式：

- **命令**：发起方主动请求接收方做决策，结果影响当前流程
- **事件**：发起方通知接收方某个事实已发生，接收方自主决定如何响应

对每条新增或变更的协作关系，明确：

| 要素 | 说明 |
|------|------|
| 发起方 / 接收方 | 谁发、谁收 |
| 命令 or 事件 | 性质 |
| 核心语义 | 这条协作表达的是什么业务事实或请求 |
| 失败语义 | 可重试 / 不可重试 / 需人工介入 |
| 幂等规则 | 重复执行如何处理 |
| 影响现有消费方？ | 是否破坏性变更 |

---

## 第六步：输出设计结果，请用户确认

```
## 域设计结果

### {域名}

**变化性质**：{能力扩展 / 业务扩展 / 边界调整 / 新建 / 合并拆分}

**不变式**：
- {约束描述}，触发场景：{场景}，违反后果：{后果}

**状态机变化**：
- 新增状态：{状态名} — {业务含义}
- 新增合法迁移：{原状态} → {新状态}（触发条件）
- 明确非法迁移：{禁止的迁移}

**协作变化**：
- {发起方} → {接收方}：{命令/事件}，{语义}，失败：{处理方式}

**影响的协作方**：{列出受影响的其他域}

---

整体域依赖关系：
{域A} ──[事件: {事件名}]──▶ {域B}
{域C} ──[命令: {命令名}]──▶ {域D}
```

## 推理样例

**场景**：用户说"支持申请退款，退款要审核"

### 事实层推导

用户描述翻译成事实：
- "退款申请"作为独立业务对象存在（不是订单的一个字段）
- 退款申请有"等待审核"这个状态
- 审核是一个需要人工介入的过程

### 主权层推导

追问：谁有权批准退款？
- 如果是业务规则决定（满足条件自动批）→ 退款域自主决定
- 如果是人工判断（客服审核）→ 退款域承载审核状态，但决策由人发出命令触发

### 约束层推导

追问：什么约束必须立即成立？
- "已发货订单才能申请退款"——这是前置条件，不是不变式（是命令的前置，不是聚合内约束）
- "同一订单只能有一笔待审核的退款申请"——这是不变式，必须立即成立

### 过程层推导

"退款要审核"意味着跨越时间：
- 状态集合：待审核 → 审核通过 / 审核拒绝
- 审核通过后触发退款执行，退款执行是另一个过程（执行中 → 已退款 / 退款失败）
- 两个过程分开设计，不合并

### 协作层推导

- 退款审核通过后，需要通知支付域执行退款 → **命令**（需要支付域做决策并执行）
- 支付域退款完成后，通知退款域更新状态 → **事件**（事实通知）

### 识别域边界

退款申请是否独立成域？
- 独立业务语言？是（"退款申请"在这里有特定含义：待审核的退款意图）
- 独立不变式？是（同一订单只能有一笔待审核申请）
- 独立演进？是（审核规则可能单独变化）

→ 退款域独立成域，承载退款申请的生命周期和审核决策主权。

## 第七步：写入文档

用户确认后，立即写入：

1. `docs/vcddd/design/{domain}/boundary.md` — 按 `reference/thinking/templates/design/{domain}/boundary.md` 格式写入
2. `docs/vcddd/design/{domain}/business.md` — 按 `reference/thinking/templates/design/{domain}/business.md` 格式写入
3. `docs/vcddd/requirements/changelog.md` — 按 `reference/thinking/templates/requirements/changelog.md` 格式追加本次设计中值得记录的内容

**如果域设计过程中发现需要回跳需求线**（触发条件见 `requirements.md` 的"与域设计的关系"章节）：
1. 停止写入，告知用户需要先澄清业务事实
2. 加载 `reference/thinking/requirements.md`，以增量更新模式执行
3. `facts.md` 更新完成并经用户确认后，回到本文件，从受影响的域重新推导

**文档写入完成后，等待用户决定是否推进实现。不主动进入实现，不提示"接下来写代码"。**

## 推理顺序与瓶颈出口

### 维度依赖顺序

域设计沿五个维度推进，维度之间有依赖关系，不能跳步：

| 维度 | 前置要求 | 跳步后果 |
|------|---------|---------|
| **事实** | 无前置，从 facts.md 读入 | — |
| **主权** | 事实已确认 | 不知道约束该写给谁 |
| **约束** | 主权已归属 | 不知道哪些迁移是非法的 |
| **流程** | 约束已明确 | 协作契约的触发时机无法定义 |
| **协作** | 流程已稳定 | 无法判断协作类型和失败语义 |

**每个维度的输出是下一个维度的输入，跳步意味着在不完整的基础上建立设计——后期必然返工。**

---

### 各维度卡住时的出口

**卡在事实层**

卡住信号：描述模糊、多义词、不同地方说法矛盾

根因：需求未澄清，business facts 不稳固

出口：停止设计，回到需求线。向用户追问，更新 `facts.md` 后再继续

---

**卡在主权层**

卡住信号：两个域都能合理地做这个决策，难以取舍

根因：边界划分有多种合理方案

出口：向用户列出两个方案及各自含义（谁拥有、谁依赖、谁会随之改变），请用户确认，不自行决定

---

**卡在约束层**

卡住信号：写出了约束，但不清楚违反后会怎样

根因：业务后果不清，约束归属的责任没想清楚

出口：追问用户两件事：①违反这条约束，谁受损？②受损后如何补救？答案能填写不变式描述，才能继续

---

**卡在流程层**

卡住信号：状态转换路径有争议，或异常路径想不清楚

根因：异常语义未澄清，或流程边界与其他域重叠

出口：先锁定正常路径，异常路径标记 `[待澄清]`，将整个流程提交用户确认后再填写异常分支

---

**卡在协作层**

卡住信号：不确定用命令还是事件，或失败语义想不清楚

根因：协作的耦合程度和接收方自主度不清

出口：追问一个判断题——"接收方的处理结果，需要影响当前流程吗？"
- 是 → 命令（同步决策，失败影响当前流程）
- 否 → 事件（异步通知，接收方自主处理）
