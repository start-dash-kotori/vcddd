### 域层与框架层分离

域业务逻辑集中在 `server/` 包，框架层是薄适配器。

```
框架适配层（HTTP 路由 / 队列消费者 / CLI）
    只做：接收请求 → 翻译参数 → 调用 server/{domain} → 序列化响应
    不做：任何业务判断、状态决策、不变式保护
        │
        ▼ 调用（把每个域当黑盒）
server/ 包（域层）
    ├── {domain-a}/                  ← 域A：最小完整可调度模块
    │   ├── aggregate.{ext}          ← 聚合根 + 不变式 + 状态机
    │   ├── commands.{ext}           ← 命令定义 + 处理
    │   ├── events.{ext}             ← 事件定义
    │   ├── repository.{ext}         ← 仓储接口（抽象）
    │   ├── {db}.repository.{ext}    ← 仓储实现（域自己维护，不外提）
    │   └── read_model.{ext}         ← 读模型查询接口
    ├── {domain-b}/
    └── {domain-c}/
```

**域内聚的价值**：每个域包含自己的业务逻辑和数据访问实现。理解或修改一个域只需读一个文件夹，不跨目录跳转，对人和 AI 都友好。

**边界规则**：
- `server/{domain}/` 内不出现框架的 import（HTTP context、DI 容器、框架类型）
- 域以第一层级划分，不按技术层（aggregates/、commands/、repositories/ 等）横切
- 仓储实现写在域目录内部，不提取到外层 `infra/` 或 `infrastructure/` 目录
- 数据库客户端由域自己持有，不在域之间共享
- 框架层不写业务逻辑，不保护不变式
- 认证/鉴权校验在框架层做，但「用户是否有权执行此业务动作」的判断归 `server/{domain}/`
- 跨域协作通过事件机制完成，不绕道框架层

### 日志规范

- 日志库：{pino / structlog / slog / tracing / …}
- 格式：结构化 JSON（所有环境统一，不用字符串拼接）

**每条日志的必填字段**：

| 字段 | 说明 |
|------|------|
| `timestamp` | ISO 8601，精确到毫秒 |
| `level` | `debug` / `info` / `warn` / `error` |
| `domain` | 产生日志的域名 |
| `action` | 当前执行的命令或事件名 |
| `trace_id` | 请求链路追踪 ID，由框架层入口处注入 |
| 业务标识 | 至少一个主体 ID（`order_id`、`user_id` 等） |

**必须记录日志的位置**：

| 位置 | 级别 | 必须包含 |
|------|------|---------|
| 命令入口 | `info` | 命令类型、关键入参摘要、发起方标识 |
| 状态迁移 | `info` | 迁移前状态、迁移后状态、触发原因 |
| 失败路径 | `error` / `warn` | 失败原因、错误分类、是否可重试、关键上下文 |
| 跨域事件发布 | `info` | 事件类型、事件 ID、目标域 |
| 外部事件消费 | `info` | 来源域、事件 ID、处理结果 |

**禁止**：只在最外层记录一条"处理失败"；日志中输出密码、token 等敏感字段；字符串拼接替代结构化字段。
