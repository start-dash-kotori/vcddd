# 需求线

需求线的职责是帮助用户把模糊的业务意图变成清晰、确认的业务事实描述。

产出是 `docs/vcddd/requirements/facts.md`——经用户确认的业务事实，是域设计的唯一输入。

## 前置条件

SKILL.md 中的理论基座必须已内化，再进入需求线。

## 入口判断：首次运行 vs 增量更新

进入需求线的第一件事，判断当前模式：

```
docs/vcddd/requirements/facts.md 是否已存在？
    │
    ├── 不存在（首次运行）
    │     → 从第一步开始，完整走完全部步骤
    │
    └── 已存在（增量更新）
          → 跳过第一步的"真相地图为空"路径
          → 直接读取已有 facts.md 建立当前真相地图
          → 从第二步开始，只处理新增或变更的内容
          → 写入时以"追加/修正"方式更新，不重写整个文件
```

## 需求线目标

1. 读取用户在 `docs/vcddd/requirements/input.md` 中写下的意图描述
2. 翻译成结构化的业务事实
3. 与用户逐步确认，直到事实无歧义
4. 写入 `docs/vcddd/requirements/facts.md`，作为域设计的输入

## 与域设计的关系

需求线和域设计线是循环关系，不是单向流水线。回跳触发条件与恢复点：

**从域设计线回跳到需求线的触发条件**：
- 域设计推导过程中发现某个事实描述有歧义（两种理解都说得通）
- 域设计推导出的边界与已有事实矛盾
- 用户在域设计过程中主动提出新的业务约束或场景

**回跳后的恢复点**：
1. 停止域设计，明确告知用户"需要先补充业务事实"
2. 回到本文件第二步，把新增内容走一遍五维翻译
3. 更新 `input.md` 和 `facts.md`，在 `changelog.md` 记录本次补充的背景
4. 返回域设计线，从**受影响的域**重新推导，不重跑整个设计流程

每次循环的锚点是 `facts.md` 的当前状态，不是对话历史。

## 输入与输出

**输入**：`docs/vcddd/requirements/input.md` — 用户用自然语言写下的意图、背景、设想

**输出**：`docs/vcddd/requirements/facts.md` — 经用户确认的结构化业务事实，是域设计的唯一输入

---

## 第一步：读取并建立上下文

同时读：
- `docs/vcddd/requirements/input.md`
- `docs/vcddd/requirements/facts.md`（如果已存在）

建立当前真相地图：
- 已确认的核心概念有哪些
- 已知的域和决策主权
- 现有的核心约束和状态机轮廓

**真相地图为空时**：从 `input.md` 的描述中提取初步轮廓，标注哪些是推断、哪些是确认。

---

## 第二步：翻译成业务事实

把用户的自然语言描述翻译成五个维度的业务事实。翻译时按顺序推导，不跳步——前一个维度不清晰，后面的维度无法准确推导。

### 维度一：事实层

**核心问题**：这件事发生后，业务世界里什么变了？

从用户描述中提取：
- 新增了什么概念（什么东西开始存在）
- 哪个已有概念的条件变了
- 哪些状态变得合法或非法
- 什么触发了什么结果

**翻译方式**：把功能描述改写成事实描述。

| 用户描述 | 翻译成事实 |
|----------|-----------|
| "用户可以申请退款" | "一笔退款申请作为独立业务对象存在于系统中" |
| "管理员可以封禁账号" | "账号存在'已封禁'这个合法状态，封禁是不可自行撤销的" |
| "支持分期付款" | "一笔订单可以对应多笔分期付款记录，每笔有独立状态" |

**卡点**：用户描述中出现"系统应该…"、"页面显示…"、"接口返回…"时，追问：这件事发生后，业务上什么不同了？

---

### 维度二：主权层

**核心问题**：谁有权承认这个变化？

对每个识别出的事实，追问：
- 这个判断由哪个域做最终裁定
- 如果两个地方都在做这个判断，会发生什么

**判断方法**：找到"如果这个判断错了，谁来承担后果"——承担后果的那个域，就是拥有主权的域。

**卡点**：同一个判断在业务上有两种合理归属时，不要自己选择，向用户提问：

> "X 这个判断，是由 A 域来决定，还是 B 域来决定？两者的区别是：[具体说明区别]"

---

### 维度三：约束层

**核心问题**：是否引入了必须立即成立的约束？

对每条新约束追问：
- 什么时候这条约束必须成立（触发场景）
- 违反了会发生什么（后果）
- 能不能延迟收敛（如果能，就不是不变式）

**高风险信号**：约束变化是最高风险的变化。出现以下情况时必须停下来确认：
- 新约束与已有约束方向相反
- 新约束的违反后果不清楚
- 用户说"一般情况下…"暗示存在例外

---

### 维度四：过程层

**核心问题**：这件事跨越时间吗？

判断标准：如果这个过程在中途停止，系统能说清楚它停在哪里吗？如果不能，就需要状态化。

出现以下任何一个，必须用状态描述，不能用调用链：
- 有等待（等用户操作、等外部系统回调）
- 有异步（触发后不立即完成）
- 有人工介入
- 有失败可能且需要恢复

**翻译方式**：

> "这个过程的状态集合是什么？每个状态允许什么发生？失败停在哪里？能否恢复？"

---

### 维度五：协作层

**核心问题**：谁需要知道这件事发生了？

对每个事实，追问：
- 哪些域需要响应这个变化
- 响应方式是命令（主动请求决策）还是事件（被动接收事实）
- 失败时怎么处理

**命令 vs 事件判断**：
- 需要对方做决策、结果影响当前流程 → 命令
- 只是通知对方某件事已经发生 → 事件

---

## 第三步：输出翻译结果，请用户确认

翻译完成后，输出给用户确认，格式：

```
## 我理解的业务事实

**新增事实**：
- {事实描述}（来自：{用户原话}）

**主权归属**：
- {判断内容} → {域名}

**新增约束**：
- {约束描述}，违反后果：{后果}

**过程状态**：
- {过程名}：{状态集合}

**协作变化**：
- {发起方} → {接收方}：{命令/事件}，{语义}

---

以下我不确定，需要你确认：
1. {问题}（选项A / 选项B，区别是：{区别}）
2. {问题}
```

**提问原则**：
- 只问真正影响业务事实或决策归属的问题
- 给出候选选项，不做开放式追问
- 一次不超过三个问题，优先问最关键的

**必须向用户确认的情况**：
- 同一判断在业务上有两种合理归属，且含义不同
- 某个过程的失败语义无法从现有信息推断
- 用户描述中存在内部矛盾

---

## 第四步：写入 facts.md

用户确认后，立即更新 `docs/vcddd/requirements/facts.md`。

按 `reference/thinking/templates/requirements/facts.md` 的格式写入。

写入规则：
- `facts.md` 始终反映当前已确认的全部业务事实，不保留历史
- 历史变迁记录在 `docs/vcddd/requirements/changelog.md`，按 `reference/thinking/templates/requirements/changelog.md` 格式追加
- 每次更新在文件头注明日期和本次更新了什么

写入完成后，才可以进入域设计。
