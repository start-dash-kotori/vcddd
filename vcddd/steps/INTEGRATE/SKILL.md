# Step: INTEGRATE — 跨域集成验证

验证各域实现之间的协作是否一致。这是系统级对抗方，检查多域的业务语义拼合。

## 前置条件

- 所有域的代码已实现并通过各自的三层审查

## 输入

- `facts.md`（业务全景）
- 各域 `business.md` + `boundary.md`
- 各域实现代码

## 输出

跨域集成验证报告

## 调度循环（控制器强制执行）

**以下是控制器的强制执行指令。每个步骤完成后立即进入下一步，不需要用户提醒。**

```
┌─ 集成验证循环开始 ─────────────────────────────────────┐
│                                                         │
│  1. 派遣集成验证 Agent                                   │
│     → 传入：facts.md + 全部域的 business.md + boundary.md│
│            + 全部域的实现代码                             │
│     → 运行全部验证检查：                                  │
│       · 契约结构（命令/事件字段匹配）                     │
│       · 端到端工作流（跨域业务流程完整性）                 │
│       · 幂等（跨域命令的幂等保护）                        │
│       · 事件顺序与并发（事件依赖关系正确）                 │
│       · 依赖方向（无反向依赖）                            │
│     → 返回：PASS / TECH_ISSUES / BIZ_ISSUES / MIXED     │
│                                                         │
│  2. 返回 PASS 后：                                       │
│     → 集成验证完成，控制器进入 REPORT 步骤                │
│                                                         │
│  3. 返回 TECH_ISSUES 后【立即】：                         │
│     → 不打断用户                                         │
│     → 对每个技术问题：                                    │
│       · 判断属于哪个域                                    │
│       · 派遣该域的 Implementer 修复                       │
│       · Implementer 修复后重新运行该域的测试              │
│     → 全部修复后【立即】重新派遣集成验证 Agent（回到 Step 1）│
│     → 循环直到 PASS 或达到 10 轮上限                     │
│                                                         │
│  4. 返回 BIZ_ISSUES 后【立即】：                          │
│     → 暂停自动化                                         │
│     → 向用户呈现场景：                                    │
│       · 当前问题描述                                      │
│       · 基于业务理解给出的建议方案                         │
│       · 影响范围                                          │
│     → 等待用户决策                                        │
│     → 用户确认后更新 business.md                          │
│     → 受影响域重新过 REVIEW-DOMAIN                        │
│     → 重新派遣集成验证 Agent（回到 Step 1）               │
│                                                         │
│  5. 返回 MIXED 后【立即】：                               │
│     → 先处理 TECH_ISSUES（Step 3，不打断用户）            │
│     → 再处理 BIZ_ISSUES（Step 4，打断用户）              │
│     → 全部处理后重新派遣集成验证 Agent（回到 Step 1）      │
│                                                         │
└─ 集成验证循环结束 ─────────────────────────────────────┘
```

**关键约束**：
- **技术问题不打断用户**：编排者直接协调 Implementer 修复
- **业务问题必须打断用户**：涉及业务决策，不能 AI 自行决定
- **修复后必须重验**：任何修复后都必须重新运行集成验证
- **不需要用户提醒**：技术问题的修复循环是自动的

## 如果卡住

读 `../../reference/engine/integration-verification.md` 获取完整验证清单。

- [ ] 业务细化问题已向用户呈现并得到回复
- [ ] 修改 business.md 后对应域已重新通过审查
- [ ] 对抗循环不超过 10 轮
