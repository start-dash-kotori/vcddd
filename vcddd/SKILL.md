# VCDDD 业务导航 Skill

VCDDD（Vibe Coding Domain-Driven Design）是一套面向 AI 辅助开发的业务建模方法论。核心命题：

> **代码可以频繁变化，但系统对业务世界的表达不能漂移。**

---

## 使用方式：渐进式加载

本 Skill 采用分层入口架构。使用时：

1. **读本文件** — 理解 VCDDD 是什么、有哪些步骤、核心规则
2. **定位到当前步骤** — 按需读 `steps/{STEP}/SKILL.md`
3. **如需具体操作细节** — 读步骤文档中引用的 reference 详细文档

不需要提前加载全部步骤的细节。每步的文档是独立的，只在你需要执行那一步时才读。

---

## 总控 Agent（Controller）

VCDDD 使用总控 Agent 协调所有步骤的执行。总控 Agent 是运行本 SKILL.md 的 AI session 自身。

### 总控的工作流程

```
用户提出需求（可能模糊，可能不完整）
    │
    ▼
[总控] 分析需求 → 判断涉及哪些步骤
    │  例："帮我分析这个项目的架构" → BROWNFIELD
    │  例："我要开发一个订单系统" → V → C → D¹ ...
    │
    ▼
[总控] 向用户确认步骤方案
    │  "我理解你需要的流程包括：① V ② C ③ D¹，是否开始？"
    │
    ▼
[总控] 逐步骤执行（或按用户确认的顺序）
    │
    ├── 读 steps/{STEP}/SKILL.md
    ├── 派遣子 Agent 执行（将步骤文档作为指令传入）
    ├── 记录执行状态和结果
    └── 判断是否进入下一步或返回用户确认
```

### 总控的职能边界

| 职责 | 非职责 |
|------|--------|
| 分析用户需求，匹配到对应步骤 | ❌ 不直接修改文件或代码 |
| 为每个步骤派遣子 Agent 并编写 Prompt | ❌ 不参与项目内部文件的实质编写 |
| 创建 Task 节点，维护进度日志 | |
| 将步骤文档作为子 Agent 的执行指令 | |
| 判断步骤间的依赖关系和执行顺序 | |

### 子 Agent 的派遣模式

每个步骤文档（`steps/{STEP}/SKILL.md`）设计为子 Agent 的完整指令。总控派遣时：

```
总控读取 steps/IMPLEMENT-DOMAIN/SKILL.md
  ↓
组成子 Agent 的 Prompt：
  "你是实现 agent。请按以下步骤执行：{IMPLEMENT-DOMAIN.md 全文}
   上下文：{当前域的 business.md / boundary.md / test-spec.md / tech-stack.md}"
  ↓
派遣子 Agent（不占用总控的上下文窗口）
  ↓
子 Agent 完成任务，返回结果
  ↓
总控记录：{步骤名} → {状态: DONE/FAILED} → {产出物路径}
```

## 步骤总览

| 调用方式 | 步骤 | 做什么 | 步骤位置 |
|---------|------|-------|---------|
| `brownfield` | BROWNFIELD — 棕地初始化 | 分析已有代码生成认知文档 | `steps/BROWNFIELD/` |
| `vision` | V — 意图视野 | 捕捉用户意图写入 input.md | `steps/VISION/` |
| `context` | C — 事实上下文 | 澄清为 facts.md（逐条确认） | `steps/CONTEXT/` |
| `domain-design` | D¹ — 域设计 | 推导边界/不变式/事件/契约 | `steps/DOMAIN-DESIGN/` |
| `dev-setup` | D² — 技术栈确立 | 确定技术栈写入 tech-stack.md | `steps/DEVSETUP/` |
| `scaffold` | D²·⁵ — 骨架生成 | 生成项目骨架代码（基础设施连接器 + 共享基设 + 框架层） | `steps/SCAFFOLD/` |
| `tdd-bridge` | TDD 桥梁 | 从 business.md 推导黑盒测试 | `steps/TDD-BRIDGE/` |
| `implement-domain` | 域实现 | 一个 agent 完整实现一个域 | `steps/IMPLEMENT-DOMAIN/` |
| `review-domain` | 域审查 | 三层审查 | `steps/REVIEW-DOMAIN/` |
| `integrate` | 跨域集成 | 验证跨域协作一致性 | `steps/INTEGRATE/` |
| `report` | 最终报告 | 汇总完成情况 | `steps/REPORT/` |

用户可直接调用任一步骤（例如"执行棕地初始化"或"帮我做域设计"），不需要从 V 开始。

## 核心规则

### 人机分工

| 阶段 | 决策者 | AI 角色 |
|------|--------|---------|
| V → C → D¹ | 用户确认门禁 | 推导、结构化、呈现请用户确认 |
| D² 技术栈 | 用户确认 | 分析推荐，由用户选择 |
| D³ 实现 + 审查 | AI 对抗循环 | Implementer ↔ Reviewer 三层对抗 |
| 跨域集成 | AI + 用户 | 技术问题 AI 修，业务问题问用户 |

### 禁止性规则

- `facts.md` 未经用户确认 → 不进入域设计
- 域设计未经用户确认 → 不进入实现
- `tech-stack.md` 不存在 → 不写代码
- **infrastructure/ 目录禁止包含任何域的表定义、仓储实现或业务逻辑**（表定义属于 `server/{domain}/`）
- 每域一个 agent 完整实现，不允许域内拆分
- 每个域必须通过三层审查（Spec → Quality → VCDDD）
- 对抗循环最多 10 轮，超 10 轮升级给用户
- 发现 `business.md` 内部矛盾 → 暂停，退回到需求线修正
- 全部域完成后 → 必须执行跨域集成验证

## 理论基座

本部分是整个 VCDDD 的思想地基。在执行任何业务分析或设计任务之前，必须先完整内化本部分。

### 一、为什么需要这套理论

#### 1.1 DDD 的历史背景

领域驱动设计（DDD）诞生于代码稀缺、重写代价高昂的时代。那个时代最大的敌人是：**业务规则被淹没在技术细节里**——事务逻辑写在 SQL 里，规则藏在 controller 里，不同模块各自维护着对同一概念的不同理解。

DDD 的历史回答是：
- 用**通用语言**让团队说同样的词
- 用**限界上下文**划定语义边界
- 用**聚合/仓储/应用服务**把规则从技术层提取出来

这套回答在当时非常有价值，但它有一个副作用：DDD 逐渐被等同于"一套战术模式"——仓储、工厂、DI、层层接口。很多团队把这些战术结构的存在当作"做了 DDD"的证明，反而忘了 DDD 要保护的对象是什么。

#### 1.2 AI 时代的新条件

AI 的加入改变了一个基本前提：**代码不再是最稀缺、最昂贵、最值得优先保护的资产**。

过去需要数周才能完成的重构，今天可以在数小时内生成初稿并多轮修正。代码的可变性大幅提高了。

但这带来了新的风险：**结构失控的速度也同步提高**。没有边界纪律的系统，在人工时代可能需要数月才逐渐腐化，在 AI 辅助开发下可能在数天内堆积出大量局部正确、整体失序的实现。

AI 时代系统的主要风险转向了另一类问题：
- 边界是否失真
- 状态是否失控
- 契约是否漂移
- 事件是否失语
- 过程是否不可治理

#### 1.3 VCDDD 的核心判断

> **代码可以变化，甚至可以频繁重写；但系统对业务世界的表达不能漂移。**

VCDDD（Vibe Coding Domain-Driven Design）不是"更轻的 DDD"，也不是"AI 版 DDD"。它是对 DDD 本体的重新界定：

**DDD 的本体不是战术模式，而是业务主权。**

真正值得被优先保护的，是：
1. **业务真相** — 系统承载的业务事实与约束
2. **语义边界** — 概念在哪里生效，在哪里失效
3. **决策归属** — 谁对哪类判断有最终解释权
4. **过程状态** — 业务过程在时间维度上的显式位置
5. **协作契约** — 跨边界协作的稳定语义
6. **验证屏障** — 防止实现偏离上述一切的机制

### 二、核心世界观（必须内化的三个判断）

#### 判断一：业务真相先于一切技术结构

设计工作的起点不是"拆模块"、不是"建表"、不是"画接口"。

起点是：**这个系统试图承载的业务事实是什么？**

业务真相回答的是：
- 哪些概念真实存在（不因技术选型变化）
- 哪些状态真实存在（业务世界允许的合法位置）
- 哪些约束必须被强制遵守（违反了谁来承担后果）
- 哪些变化被系统承认（什么触发了什么结果）

**错误信号**：一开始就围绕页面、接口、数据库字段来讨论——这样得到的是技术清单，不是业务真相。

#### 判断二：系统是决策与协作网络，不是技术模块的组合

一个限界上下文的意义，不在于它拥有多少类，**而在于它是否拥有明确的语义主权**。

一个聚合的意义，不在于它包含多少对象，**而在于它是否守住了必须同步成立的不变式**。

一个事件的意义，不在于它是否使用了消息系统，**而在于它是否忠实表达了一个已经发生的业务事实**。

一个过程的意义，不在于它是否跑通，**而在于它是否具备可观察、可恢复、可重试、可终止的治理能力**。

#### 判断三：战术结构是手段，不是目的

仓储、工厂、DI、策略模式——这些构件在特定场景下有用。

但它们的存在不等于系统有了 DDD。**把战术结构的存在误当成边界纪律的证明，是 DDD 最常见的误用。**

VCDDD 不反对战术，VCDDD 反对把战术当成本体。

### 三、思考模型

有了世界观，还需要一套**推理方式**。思考模型不是步骤，而是面对任何业务问题时大脑自动运转的逻辑。

#### 3.1 从"事实"切入，而不是从"功能"切入

大多数人面对需求，第一反应是"要做什么功能"。VCDDD 的切入点不同：
> **这件事发生后，世界的哪个部分变了？那个变化是什么？**

功能是手段，事实是目的。"用户下单"不是目的，"一笔待履约的交易承诺存在于世界中"才是目的。一旦从事实切入，以下问题会自然浮现：
- 这个事实由谁来承认？（决策归属）
- 这个事实在什么条件下成立？（不变式）
- 这个事实会触发哪些其他变化？（事件与协作）

从功能切入，得到的是接口列表。从事实切入，得到的是业务模型。

#### 3.2 边界是"谁说了算"，不是"代码放在哪"

每当看到一个概念、一个动作、一个状态，首先要问的不是"它属于哪个模块"，而是：
> **谁对这件事有最终解释权？**

这个问题逼迫思考的是**主权**，而不是归属。"订单状态"放在 order 模块是归属，"只有订单域才能判断一笔订单是否可撤销"是主权。两件事看起来相似，但前者只是组织代码，后者是在保护决策边界。

主权的边界比代码的边界更稳定。代码可以重组，但"谁有权做这个判断"通常反映的是业务本质，很少因为技术演进而改变。

所以划边界的正确问法是：**如果两个地方都在做这个判断，会发生什么？** 如果答案是"语义冲突"或"谁也说不清楚"，边界就划错了。

#### 3.3 过程是时间维度上的事实，必须被显式承载

很多系统的 bug 不是来自逻辑错误，而是来自**过程没有被当作一等公民**。

一个支付流程，如果只存在于"调用支付接口 → 等回调 → 更新状态"这条调用链中，那么：
- 回调没来，谁负责？
- 中途崩溃，从哪恢复？
- 重试了，幂等怎么保证？
- 超时了，算成功还是失败？

这些问题在调用链里无法回答，因为调用链只描述"顺利时怎么走"，不描述"过程在时间轴上处于什么位置"。

VCDDD 的思考方式是：**任何跨越时间的业务过程，都必须先问"它的状态集合是什么"**。状态一旦被显式定义，失败、重试、恢复、审计才有落点。

判断一个过程是否需要状态化的标准很简单：**如果这个过程在中途停止，系统能说清楚它停在哪里吗？** 如果不能，就需要状态化。

#### 3.4 不确定性靠契约解决，不靠实现耦合

跨边界时，永远存在不确定性：对方什么时候响应、失败了怎么算、字段含义是否一致。

解决这种不确定性有两条路：
1. **共享实现**：直接访问对方的数据库、调用对方的内部模型——确定性高，但耦合也高，边界实际上消失了
2. **定义契约**：双方只依赖一份稳定的语义协议，内部各自演进——耦合低，但需要显式维护

VCDDD 只认第二条路。原因不是工程洁癖，而是现实：**共享实现的代价在边界消失那一刻不明显，在一年后演进时才彻底爆发。**

所以每当出现跨边界协作，思考方式应该是：**这里的不确定性，现在靠什么在保证？是隐式假设，还是显式契约？** 隐式假设迟早会成为 bug 的温床。

#### 3.5 四条推理主轴的关系

这四条不是并列规则，而是一套相互推导的逻辑：

```
从"事实"切入
    → 识别出谁的事实、谁说了算
        → 边界是主权，不是代码归属
            → 跨边界时，过程和协作都必须显式化
                → 不确定性靠契约，不靠耦合
```

当你拿到任何问题——一个新需求、一个边界冲突、一个设计决策——都可以从这四条主轴切入，逐步推导出应该保护什么、应该怎么划分、应该怎么协作。

### 四、核心概念体系

以下概念是可操作的工具，而不是术语表。每个概念都有一个核心问题——那个问题才是使用这个概念时真正要回答的东西。

#### 4.1 业务真相（Business Truth）

**核心问题**：这个系统对业务世界做出了哪些必须成立的承诺？

业务真相包括：
- **核心概念**：系统承认哪些业务对象真实存在
- **合法状态**：这些对象可以处于哪些状态
- **不变式**：哪些约束必须在任何时候立即成立
- **变化触发点**：什么事件会让状态发生迁移

**判断技巧**：问"如果技术栈整体替换，这条规则还成立吗？"成立的才是业务真相，不成立的是实现细节。

#### 4.2 通用语言（Ubiquitous Language）

**核心问题**：这个边界内，所有人（包括代码）是否用同一套词描述同一件事？

通用语言不是术语表，它是**命名契约**。它要求：
- 业务动词统一（创建/生效/冻结/结算/撤回）
- 业务名词统一（订单/账单/资质/会话/任务）
- 状态名称统一
- 事件名称统一
- 错误名称统一

而且这套词必须在文档、接口、代码、日志中全程一致。

#### 4.3 限界上下文（Bounded Context）

**核心问题**：通用语言在哪里生效？超出这个边界，这套词就不再适用或含义不同。

限界上下文不是代码包，不是目录结构。它是**语义主权的作用域**。

设计时要回答：
- 这个范围内是否存在稳定且独立的业务语言？
- 这个范围内是否有自己的关键判断？
- 这个范围内的对象是否需要独立演进？
- 与外部只能通过协作，不能共享内部模型？

**拆分判断**：多个部分虽然处理同一个名词，但如果它们关心的状态、判断、约束或参与该名词的方式不同，那么它们通常不应被强行压进同一上下文。

**合并判断**：两个模块只是技术上分开，但业务语义、状态、决策和协作角色都完全一致 → 不应拆成多个上下文制造空边界。

#### 4.4 决策边界（Decision Boundary）

**核心问题**：对这类业务判断，谁有最终解释权？

每个关键业务判断必须有**唯一归属**。

决策边界要明确：
- 哪个上下文对哪类状态变更有批准权
- 哪些规则属于同步决策，不能外包给"后续处理"
- 同一判断不能有两个归属

#### 4.5 聚合（Aggregate）与不变式（Invariant）

**核心问题**：哪些约束必须在一个原子边界内同时成立？

聚合不是"把相关对象装在一起的容器"，它是**最小足够同步的一致性边界**。

设计聚合时问：
- 哪些判断必须立即完成？
- 哪些约束不能等到后续再校正？
- 哪些内部状态修改不应对外暴露？

**大小原则**：聚合应该尽量小，但不能小到守不住关键不变式；聚合不应大到把本该异步协作的关系强行装入同步边界。

**不变式分类**：
- **立即成立**：任何时候都必须成立，进聚合
- **可延迟收敛**：允许短暂不一致，通过事件异步修正

#### 4.6 命令（Command）、事件（Event）与状态（State）

三者共同构成系统对动态业务的表达：

| 概念 | 表达的是 | 关键边界 |
|------|---------|---------|
| **命令** | 业务意图，触发决策 | 不等于立即完成；不是 CRUD，是业务动词 |
| **事件** | 已经发生的业务事实 | 不等于通知；必须可在未来重放；表达事实，不是回调 |
| **状态** | 业务过程在时间轴上的位置 | 不等于数据库字段；首先是业务语义，其次才是存储 |

#### 4.7 过程状态（Process State）

**核心问题**：这个业务过程在时间维度上，现在处于什么位置？允许什么发生？

必须显式定义：
- 过程从什么状态开始
- 中间经过哪些合法状态
- 每个状态允许什么动作（及其前置条件）
- 什么条件导致进入失败、终止或待人工处理
- 哪些失败可以重试，哪些必须终止

#### 4.8 协作契约（Collaboration Contract）

**核心问题**：跨边界协作时，双方的业务协议是什么？

契约包括：命令/事件的结构（字段与语义）、错误语义（可重试、不可重试、人工介入）、幂等规则、时间要求、演进规则。

#### 4.9 上下文映射（Context Map）

**核心问题**：多个上下文之间，集成关系是什么？谁依赖谁，方向如何？

常见关系类型：客户-供应（Customer-Supplier）、防腐层（Anti-Corruption Layer）、共享内核（Shared Kernel）、遵奉者（Conformist）。

#### 4.10 读模型（Read Model）

**核心问题**：这个视图是为了展示/查询/追踪，不是为了决策。

读模型不承担业务决策责任。设计时要确认谁需要看见这些信息、是为了判断还是展示。

#### 4.11 验证屏障（Validation Barrier）

**核心问题**：如何防止实现在持续演进中偏离以上所有定义？

验证屏障类型：类型与 Schema 校验、不变式校验、状态机校验、幂等校验、事件顺序校验、契约测试。

### 五、自动化执行层

#### 为什么 D²/D³ 可以自动化

V→C→D¹ 必须有人工门禁，因为**业务真相只能由人类确认**。但 D²/D³ 是规则驱动的执行工作——输入是已确认的文档，输出是符合这些文档的代码。

只要文档是精确的（D¹ 已确认），实现就可以是自动的。

#### 自动化的运作方式

D¹ 确认后，编排者加载 `reference/engine/automated-execution.md`，自动完成：

1. **D²-auto**：遵循决策顺序——用户有明确要求则遵循；有现有代码则分析后向用户呈现确认；无代码则提供方案供用户选择。仅用户声明"不管"时 AI 自行决定
2. **D²·⁵-auto（SCAFFOLD）**：根据 tech-stack.md 生成项目骨架代码（基础设施连接器 + 域共享基设 + 框架层），确保 infrastructure 不含域逻辑
3. **域依赖排序**：构建域依赖图，拓扑排序确定实现顺序
4. **逐域执行**：每域依次 TDD 桥梁 → Implementer 完整实现 → 三层审查对抗循环
5. **跨域集成验证**：验证跨域契约一致，技术问题自动修，业务问题问用户

#### 辩论的 Ground Truth

实现 agent 和审查 agent 之间的辩论不是主观判断，而是以文档为准：
- Spec Compliance 辩论只能引用 business.md 和 boundary.md
- Code Quality 辩论只能引用 tech-stack.md
- 任何一方引用"感觉"或"最佳实践"均无效
