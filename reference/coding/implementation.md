# 实现线

实现线的职责是把已确认的域文档翻译成具体的技术实现设计，并指导代码编写。

**输入**：
- `docs/vcddd/design/{domain}/boundary.md`
- `docs/vcddd/design/{domain}/business.md`
- `docs/vcddd/tech-stack.md`（按需读取，见第零步；不存在时先走技术确立线）

**输出**：
- `docs/vcddd/design/{domain}/implementation.md`
- 代码

**触发条件**：由用户主动发起，不由模型主动进入。

---

## 方法论：从域文档到实现设计

### 第零步：技术设定检查（入口强制执行）

进入实现线做的第一件事，只有两个分支，没有例外：

```
检查 docs/vcddd/tech-stack.md 是否存在
    │
    ├── 存在
    │     → 读取 tech-stack.md 全文
    │     → 不加载 tech-setup.md，不执行任何识别流程
    │     → 直接进入第一步
    │
    └── 不存在
          → 停止实现线
          → 加载 reference/coding/tech-setup.md
          → 完整执行技术确立流程，直到 tech-stack.md 写入完成
          → 写入完成后回到本文件，从第零步重新进入（此时文件已存在，走上方分支）
```

**tech-stack.md 存在时，tech-setup.md 的全部内容不进入上下文。**

读取 tech-stack.md 后，在整个实现过程中始终遵守其中规定的：
- 命名约定（变量、函数、类、文件、事件、命令的命名风格）
- 目录结构约定（新文件放在哪里）
- 使用的库和工具（不引入 tech-stack.md 未记录的新依赖，如需引入先向用户确认并更新文档）
- 日志规范（库、格式、必填字段、必须打日志的位置）
- 测试约定（框架、文件位置、命名方式）
- 错误处理方式

**发现代码与 tech-stack.md 冲突时，以 tech-stack.md 为准修改代码。**

---

### 第一步：读域文档，建立实现地图

同时读 `boundary.md` 和 `business.md`，提取：

- 所有命令及其前置条件、正常路径、失败路径
- 所有不变式及其归属聚合
- 完整状态机：合法迁移、非法迁移、失败语义
- 跨域协作：命令和事件的方向与契约

这张地图是实现设计的全部依据。**代码必须覆盖文档里的每一条路径，包括失败路径。**

---

### 第二步：设计模块结构

根据域文档里的职责，设计代码目录结构。每个文件对应一个明确的业务职责，不按技术层（controller/service/repository）组织，按业务职责组织。

将模块结构设计结果写入 `docs/vcddd/design/{domain}/implementation.md`，按 `reference/thinking/templates/design/{domain}/implementation.md` 格式写入"模块结构"部分，说明每个文件承载的业务职责。

---

### 第三步：逐命令设计实现方案

对每个命令，设计：

1. **正常路径**：步骤和关键技术手段
2. **所有失败路径**：如何捕获、如何处理、是否回滚、错误响应格式
3. **幂等策略**：幂等键的设计和检测方式（必填，不可跳过）
4. **事务边界**：哪些操作在同一事务内，哪些不在，选择这个边界的原因（必填，不可跳过）
5. **异步处理**：如果有异步逻辑，异步的边界在哪里，失败时如何重试

**业务文档里的每条命令处理路径，实现设计里必须有对应的分支，不允许遗漏失败路径。**

---

### 第四步：记录技术决策

每个重要的技术选择都要记录：选了什么、为什么选、否定了什么方案、否定的原因。

**没有记录被否定方案的技术决策是不完整的**——下次遇到同样问题会重新讨论已经决定过的事情。

---

### 第五步：遇到文档未覆盖的情况

实现过程中发现业务文档未覆盖的边界情况时：

1. **不自行补充兜底逻辑**，不假设用户的意图
2. 停下来，向用户描述这个情况，给出候选处理方案，请用户确认
3. 用户确认后，先更新 `business.md`，再继续实现

**未经讨论的兜底方案与用户实际需求大概率不符，不写。**

---

## 强制规范

以下规范不论技术栈，必须遵守。

---

### 命名规范

**命名必须与通用语言一致。**

- 代码里的类名、方法名、变量名、事件名、错误码，必须与 `boundary.md` 和 `business.md` 里的业务词汇完全一致
- 禁止用技术词汇替换业务词汇：不用 `create` 替代 `place`（下单），不用 `update` 替代 `confirm`（确认）
- 禁止同一概念在代码里出现多个名字（如 `order` vs `purchase` vs `trade`）
- 状态名称必须与状态机文档一致，不自行缩写或改写

**判断方法**：把代码里的名字念给业务方听，如果对方需要翻译才能理解，命名就错了。

---

### 日志规范

**每个关键业务动作必须有日志，失败路径必须有日志。**

必须记录日志的位置：
- 命令入口：记录命令类型、关键入参、发起方标识
- 状态迁移：记录迁移前状态、迁移后状态、触发原因
- 失败路径：记录失败原因、失败时的关键上下文、是否可重试
- 跨域事件发布：记录事件类型、事件ID、目标域
- 外部事件消费：记录事件来源、事件ID、处理结果

日志字段要求：
- 必须包含业务标识（如订单ID、用户ID），不只记录技术标识
- 必须包含时间戳
- 失败日志必须包含错误分类（业务拒绝 / 系统异常 / 外部依赖失败）

**不允许只在最外层记录一条"处理失败"的日志**——必须在失败发生的位置记录，保留现场。

---

### 幂等规范

**每个命令必须有明确的幂等策略，不允许跳过。**

幂等键设计原则：
- 幂等键必须由调用方生成并传入，不由服务端生成
- 幂等键必须能唯一标识一次业务意图（不是请求ID，是业务动作ID）
- 幂等检测必须在事务内完成，不允许先检测后执行的非原子方案

幂等响应规则：
- 重复请求命中幂等键时，返回与首次相同的结果，不报错
- 幂等命中必须记录日志（区分"首次执行"和"幂等命中"）

---

### 事务边界规范

**事务边界必须与聚合边界对齐。**

- 一个事务只修改一个聚合的状态
- 跨聚合的操作必须通过事件异步协调，不放在同一事务内
- 事务内不发起外部调用（HTTP、消息队列发布等）；外部调用在事务提交后执行

违反这条规范的典型信号：
- 一个事务里同时写两张核心业务表
- 事务内调用了外部 API

---

### 错误处理规范

**失败路径必须显式处理，不允许只靠全局异常捕获兜底。**

错误分类：
- **业务拒绝**：前置条件不满足、不变式冲突——返回明确的业务错误码和原因，不重试
- **系统异常**：数据库故障、内存溢出——记录日志，根据幂等策略决定是否重试
- **外部依赖失败**：第三方服务不可用——明确是否可重试，重试策略必须有上限

错误响应格式：
- 必须包含错误码（业务语义，不是 HTTP 状态码）
- 必须包含错误描述（业务语言，不是异常堆栈）
- 业务拒绝必须说明原因（"订单已发货，不可撤销"，不是"操作失败"）

---

### 状态机保护规范

**状态迁移必须通过聚合内的方法执行，不允许直接赋值状态字段。**

- 非法迁移必须在代码层面被拒绝，不依赖数据库约束或调用方自觉
- 每次状态迁移必须同步产出对应的领域事件（即使暂时没有消费方）

---

### 文档同步规范

**代码必须与域文档一致，出现偏差以文档为准修改代码，不以代码为准修改文档。**

实现过程中发现文档与实际需求不符时：
1. 停止编码
2. 回到需求线或域设计线，修正文档
3. 文档经用户确认后，再继续编码

---
