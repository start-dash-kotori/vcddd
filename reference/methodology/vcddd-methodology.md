# VCDDD 方法论

> 版本 1.0 | 2026-04

---

## 概览：两层含义

VCDDD 同时承载两层概念，它们不是替代关系，而是上下包含关系。

```
┌─────────────────────────────────────────────────────┐
│              第二层：五步工作方法论                    │
│   V → C → D¹ → D² → D³                             │
│   Vision · Context · Domain · Dev Setup · Develop   │
├─────────────────────────────────────────────────────┤
│              第一层：理论基座                          │
│   Vibe Coding + Domain-Driven Design                │
│   AI 时代对 DDD 本体的重新界定                        │
└─────────────────────────────────────────────────────┘
```

---

## 第一层：理论基座

**VCDDD = Vibe Coding Domain-Driven Design**

这是一个对 DDD 本体的重新定义，而不是对 DDD 的修补或包装。

### 为什么需要这个新定义

传统 DDD 诞生于代码昂贵、重写代价高的年代，它的保护对象是「实现」——通过仓储、聚合、应用服务等战术模式把业务规则从技术细节里拯救出来。

AI 时代改变了一个基本前提：**代码不再是最稀缺的资产**。重写的成本大幅下降，但结构失控的速度也同步加快。系统真正需要被优先保护的对象已经转移：

| 传统 DDD 保护的 | VCDDD 保护的 |
|---|---|
| 内部实现结构 | **业务真相** |
| 代码的整洁分层 | **语义边界** |
| 战术模式的规范使用 | **决策归属** |
| — | **过程状态** |
| — | **协作契约** |
| — | **验证屏障** |

### 核心命题

> **代码可以变化，甚至可以频繁重写；但系统对业务世界的表达不能漂移。**

VCDDD 不反对战术（仓储、工厂、DI），但反对把战术误当作 DDD 的本体。真正的 DDD 本体是**业务主权**，而非战术仪式。

### Vibe Coding 的定位

「Vibe Coding」不是随意编码，而是在 AI 辅助开发的语境下，**用直觉与意图驱动代码生成，但必须以业务真相和边界约束为锚点**。它承认代码的高可变性，同时坚持业务模型的稳定性。两者不矛盾：正因为代码可以随时重写，业务真相才更需要被提前、独立地建立和保护。

---

## 第二层：五步工作方法论

VCDDD 的五个字母对应一个工程项目从零到可交付代码的完整推进路径。每一步都有明确的产出物、完成标准，以及禁止穿越的前置门禁。

```
V ──► C ──► D¹ ──► D² ──► D³
意图  事实   域设计  开发基础  代码实现
```

---

## V — Vision（意图视野）

### 这一步在做什么

把用户的自然语言描述——无论是模糊的想法、口头表达还是产品需求——转化为**可操作的原始输入**。

Vision 不是需求规格说明书，不是功能列表，更不是技术方案。它是人类意图的第一次结构化表达，目的是让后续所有分析工作有一个确定的出发点。

### 产出物

`docs/vcddd/requirements/input.md`

包含：
- 用户原始意图的忠实描述（不做删减或解释）
- 初步识别出的核心业务场景（3-7 个）
- 明确的产品边界（做什么、不做什么）
- 设计原则或价值取向（如果用户有明确表达）

### 完成标准

- 用户确认：「这是我想做的东西」
- 没有任何技术词汇混入意图描述
- 没有进行任何域分析或架构判断

### 典型错误

- 把「用户要一个订单系统」直接推导为「需要 Order 聚合」——这是跳越了 C 和 D¹
- 在 Vision 阶段引入 API 设计、数据库方案讨论
- 把功能列表当作意图描述（「系统应该支持 A、B、C」不等于意图）

### 核心问题

> **这个人/团队在试图解决世界上的哪个真实问题？他们认为什么算是成功？**

---

## C — Context（事实上下文）

### 这一步在做什么

把 Vision 中的意图，通过系统性澄清，转化为**经用户确认的业务事实**。

Context 是业务真相的制造车间。它的任务是把模糊转化为清晰，把假设转化为已确认，把感觉转化为可操作的约束。这一步的产出直接决定了后续域设计的输入质量——垃圾进，垃圾出。

### 产出物

`docs/vcddd/requirements/facts.md`，包含：

- **业务事实**：系统必须承认的客观存在（哪些概念、哪些状态）
- **业务约束**：必须成立的规则（不因技术选型而改变）
- **过程状态机**：关键业务流程的状态集合与合法迁移
- **通用语言词表**：边界内稳定使用的核心词汇（动词 + 名词 + 状态词）
- **明确排除项**：经确认不在本次范围内的内容

### 完成标准

- **用户逐条确认** facts.md 中每一条事实
- 通用语言词表已建立且无歧义
- 核心流程的状态机已经画出并确认
- 没有任何「我们假设」或「应该是」的表述残留

### 验证技巧

对每一条写入 facts.md 的内容，问：

> **如果技术栈整体替换，这条规则还成立吗？**

成立的是业务真相，不成立的是实现细节——不属于 facts.md。

### 典型错误

- 把「用户有登录功能」写入 facts（这是实现细节，不是业务事实）
- 没有经过用户逐条确认就进入域设计
- 通用语言词表留有同义词（`order` vs `purchase` vs `trade` 三词并存）

### 核心问题

> **这个系统必须对哪些业务事实做出承诺？如果这些事实不成立，一切就毫无意义。**

---

## D¹ — Domain Design（域设计）

### 这一步在做什么

以**确认过的 facts.md** 为唯一输入，推导出系统的**域边界、决策归属、不变式设计、事件与契约结构**。

Domain Design 是整个方法论中技术含量最高的一步，也是最容易被跳过的一步。它的产出是架构的「真相地图」——代码写什么、不写什么、谁管什么，都从这张地图里读出来，而不是从程序员的直觉里冒出来。

### 产出物

每个识别出的域都有三个文件：

```
docs/vcddd/design/{domain}/
├── boundary.md       ← 能力边界 + 责任边界 + 上下文映射关系
├── business.md       ← 域内核心概念 + 不变式 + 状态机 + 事件设计
└── implementation.md ← 此步结束前可以留空，等 D² 完成后再写入
```

### 完成标准

每个域的 boundary.md 和 business.md 必须能回答：

1. **能力边界**：这个域能做什么、不能做什么？
2. **决策归属**：哪些业务判断只能由这个域做出？
3. **核心不变式**：必须在聚合边界内同步成立的约束是什么？
4. **状态机**：关键业务对象有哪些合法状态，迁移条件是什么？
5. **对外事件**：这个域会发布哪些业务事实（已发生的，不是通知）？
6. **协作契约**：与其他域的协作如何定义，依赖方向如何？

用户确认所有域设计后，方可进入 D²。

### 典型错误

- 从技术模块（controller/service/repo）反推域边界
- 在域设计阶段开始讨论数据库表结构
- 把「调用关系」当作「协作契约」（前者是实现，后者是语义协议）
- 同一个业务判断出现在两个域的 business.md 中（决策主权未定义）

### 核心问题

> **谁对哪类业务判断有最终解释权？如果两个地方都能做这个判断，谁说了算？**

---

## D² — Dev Setup（开发基础）

### 这一步在做什么

在业务设计确认之后，建立代码工程的**技术基座**。这不是选型会议，而是把选型结果**写成文档约定**，让后续所有代码生成都有章可循。

Dev Setup 的核心产物是 `tech-stack.md`——它是实现线的最高权威文档，任何代码都必须在它的约束范围内生成，任何偏离都必须先更新它再写代码。

### 产出物

`docs/vcddd/tech-stack.md`，包含：

- **分层架构定义**：目录结构、层职责、依赖方向
- **技术选型锁定**：语言版本、框架、核心依赖库（版本号精确）
- **幂等约定**：所有跨边界动作如何保证幂等
- **事务边界约定**：哪些操作在同一事务，哪些不允许跨事务
- **日志与可观察性约定**：结构化日志格式、关键埋点位置
- **测试策略约定**：单元/集成/契约测试的边界与工具
- **命名约定**：文件名、类名、方法名、事件名的命名规则

### 完成标准

- 所有约定都已写入 tech-stack.md，没有「口头约定」残留
- 每个约定都有**为什么**，而不只是是什么
- 实现 D³ 之前，工程骨架（脚手架 + CI 基础配置）已经跑通

### 典型错误

- tech-stack.md 只有技术名称，没有约定细节
- 把「技术选型」和「架构约定」混为一谈（前者是工具，后者是规则）
- 在 D² 没有完成时开始写业务代码

### 核心问题

> **每一行将要生成的代码，应该落在哪个层、遵循哪些规则、如何与其他代码协作？这些规则是否都已经被文档化？**

---

## D³ — Develop（代码实现）

### 这一步在做什么

在 D¹ 的域设计和 D² 的技术约定双重指引下，**生成真正的工程代码**。

这是唯一一步真正在写代码的步骤。但它不是「随便写」，而是「以文档为准写」。每一个实现决策都必须能在 boundary.md、business.md 或 tech-stack.md 中找到依据。

### 实现原则

**代码伺候文档，文档不伺候代码。**

- 代码实现必须与 business.md 中的状态机、命令、事件、不变式一一对应
- 代码命名必须与通用语言词表（facts.md）完全一致
- 代码依赖方向必须与 boundary.md 中的上下文映射一致
- 遇到 tech-stack.md 未覆盖的新情况，必须先更新 tech-stack.md，再写代码
- 遇到 business.md 与实现有冲突，必须先澄清业务真相，而不是凭直觉选一个

### 产出更新

- 代码本身（工程目录）
- 每个域的 `implementation.md`（记录技术设计决策，而非重复代码）
- `tech-stack.md` 的必要版本更新

### 完成标准门禁（逐层检查）

```
代码可以合并的条件：
  ✅ 状态机转换都有验证屏障（类型/Schema/状态校验）
  ✅ 跨域命令/事件都有幂等保护
  ✅ 关键不变式有聚合边界保护（不能被外部绕过）
  ✅ 契约测试存在（跨域协作有测试锁定语义）
  ✅ 命名与通用语言词表一致（不允许同义词出现）
```

### 典型错误

- 看到 facts.md 不够用，直接在代码里「顺手加逻辑」而不回头更新文档
- 发现两个域的契约不清晰，用「先跑通再说」代替「先澄清再写」
- 以「AI 生成的代码本身就是对的」为由，跳过验证屏障
- 把 implementation.md 当作复制代码的地方，而不是记录决策的地方

### 核心问题

> **这行代码是在实现哪条业务真相？它在哪个文档里有依据？**

---

## 三个关键门禁

VCDDD 依赖严格的单向推进。每个步骤之间存在不可绕过的门禁：

```
V ──[用户确认 input.md]──► C ──[用户确认 facts.md]──► D¹
                                                       │
                              ┌────────────────────────┘
                              [用户确认所有域的 boundary + business]
                              │
                              ▼
                             D² ──[tech-stack.md 完成]──► D³
```

| 门禁 | 前置条件 | 违反后果 |
|---|---|---|
| C 之前 | input.md 经用户确认 | 对错误意图做了正确设计 |
| D¹ 之前 | facts.md 经用户逐条确认 | 基于假设而非事实建立域模型 |
| D² 之前 | 所有域设计经用户确认 | 代码和业务模型从一开始就对不上 |
| D³ 之前 | tech-stack.md 存在且完整 | 代码各自为政，架构规则只存在于记忆中 |

---

## VCDDD 的双层结构总结

| 层次 | 名称 | 定位 | 关键命题 |
|---|---|---|---|
| 第一层 | 理论基座 | Vibe Coding + DDD 的重新界定 | 保护业务主权，而非保护实现结构 |
| 第二层 | 工作方法论 | V→C→D¹→D²→D³ 五步框架 | 每步有产出物，有门禁，不可跳跃 |

两层互为支撑：
- 没有第一层，五步只是形式，执行者不知道「为什么不能跳」
- 没有第二层，理论只是白皮书，实践者不知道「怎么一步步做」

**VCDDD 的终极目标**：

> 在 AI 可以随时重写代码的时代，让系统对业务世界的表达保持稳定、可治理、不漂移。
